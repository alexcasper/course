 Database Systems Development Lifecycle

 

Software applications developed over the last few decades have generally required constant maintenance such as correcting programming faults (i.e. “bugs”), implementing new user requirements, or to modify the software in its entirety in order for it to run successfully on new or upgraded platforms or hardware.

The effort that has been spent on such maintenance can be costly in both time and finances, in addition to the absorption of other resources.

This had a severe impact on many major software projects making them late to deliver, over budget, unreliable, increased difficulty in maintenance and inefficient performance of the final product.

As far back as the late 1960s, worries of a “software depression” ensued.

Three key major reasons as to why these software projects failed were attributed to:

  • Lack of information in relation to specific requirements.
  • Lack of information in relation to the appropriate development methodology.
  • The inability to split the design into manageable components.

This therefore led to the decision to implement a more structured approach to developing software applications known as the Information Systems Lifecycle (ISLC).

In a similar manner, we now have the Database Systems Development Lifecycle (DSDLC) which involves the following:

Database Planning

These are management activities which allow stages of the DSDLC to be efficiently and effectively realized.

The planning must also be integrated with the overall Information System strategy of the business.

The database planning should also include the development of agreed standards that will dictate how:

  • The data is to be collected and processed.
  • What the format of the data will be and how to specify it.
  • What essential documentation will be required.
  • How the design and implementation of the database will be achieved.

System Definition

This involves the agreeing of the scope and boundaries of the database system and the major user views, which define what will be required of a database system from the perspective of a member of staff. 

The user views must be identified early to avoid any major issues, such as not accounting for senior members of staff.

The user views can also assist in the development of complex database systems in which the requirements can be split into smaller manageable sections.

Requirements Collection and Analysis

This is a very important part of the cycle because it involves the collection, processing and analysis of the information pertaining to the area of the business that will be supported by the database system.

Information concerning how the database is to be used (and by who) can be gathered by a specially constructed questionnaire.

Database Design

This is the process of creating a design for a database system that will support the mission statement and mission objectives of the business.

For the construction of a successful data model, we need to understand what entities, relations and attributes will be required.

The three distinctive phases required for the design of a database are:

  • Conceptual database design
  • Logical database design
  • Physical database design

DBMS Selection

This is required to support the database system.

However, if one is already “available” then this step can be skipped.

It can be performed at any stage prior to the logical design as long as the appropriate information has been supplied in terms of the system requirements. The key steps involved in choosing a DBMS are:

  • To define the Terms of Reference of study.
  • Shortlist two or three (available) products.
  • Evaluation of the products.
  • Recommendation of the selection of the product accompanied by a detailed report.

Application Design

This involves the design of the user interface and the application programs that will employ and process the database.

This goes hand-in-hand with the database design.

There are two important activities:

  • Transaction design
    • Actions or series of actions that will be carried out by a single user or application program which will access or change the content of the database.
  • User interface design
    • Deciding on the layout of the interface including a meaningful title, visually appealing layout, along with clear and concise field labels.

Prototyping

This is a working model of a database system. It can assist in identifying the features of a system that work very well, or that under perform.

Implementation

This is the actual physical realization of the database and application program designs.

Data Conversion and Loading

This involves the transference of any existing data into a new database and applying any appropriate conversions to the data format or type.

It also involves converting any existing applications to be able to run on the new database.

Testing

The purpose of testing is to run the database system so that any errors can be discovered.

This involves carefully planned procedures and realistic data.

The testing will not be able to highlight the absence of faults, but it will be able to show that software faults are present.

Operational Maintenance

This is the process of monitoring and maintaining the database system after it has been completely installed.

During the performance monitoring, if it is found to fall then retuning and reogranisation of the database may be initiated. 

There will also be the requirement to maintain and upgrade the database application and to also incorporate any new requirements into the database application.

 

We can see how these link together in the figure below.

The flow diagram demonstrating the Database System Development Lifecycle (DSDLC).

The starting point is the “Database planning” which leads into the “System definition” and then the “Requirements Collection and Analysis” (user preferences and requirements input).

The corresponding output can either move to the design of the database and/or the design of the software application.

From either output, there is the option to develop a prototype whose output can feed into additional requirements, or to implement the build of the database and/or software application.

The efficiencies of data conversion and loading, and then formal testing can feed into the “operational maintenance”, the output of this feeding back into the database planning should updates/upgrades be required.

From this, the cycle repeats itself.

 

Database Design Step in the Database Systems Development Lifecycle

In the Database Systems Development Lifecycle, where we are concerned with the database design (conceptual, logical and physical), we have additional yet very important steps to include/consider (click on the tabs):

1. Designing file organisations and indexes

This involves understanding the typical workload that the database must support.

  • Analysing transactions by assessing the performance criteria of the transactions that will run on the database, leading to identification of parts of the database that may cause performance problems.
  • Choosing the efficient type of file organization for each base relation, and documenting the choice.
  • Choosing the index.
  • Estimation of the disk space requirements for the database.

2. Designing the user views

This may be based on information that was collected from a questionnaire or some other form of ‘Requirements Collection and Analysis’.

  • For example, different staff levels/grades may require different views. Documentation is vital.

3. Designing the security mechanisms

This is specified by the users. Documentation is vital.

4. Controlling data redundancy.

This is to see if system performance can be improved.

Some examples include:

  • Combining one-to-one relationships.
  • Introducing repeating groups.
  • Creating extract tables.
  • Partitioning relations. 

5. Monitoring and tuning the operational system

The objective is to improve performance of the system by correcting for an inappropriate design decisions, or to reflect any changing requirements.

It is especially import to understand how the various hardware components interact with and affect the performance of the database.
