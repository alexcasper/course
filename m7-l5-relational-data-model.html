The Relational Model

Most of the modern commercial systems tend to be based on the relational data model approach.

The relational database management system (RDBMS) is now the most popular and thus dominant data processing software employed today.

The model had been first proposed when Codd presented in a paper “A relational model of data for large shared databanks” back in 1970. This paper has now become generally accepted as the landmark in database systems.

The objectives of the relational data model were specified as:

  • To permit a high degree of data independence:
  • To provide substantial grounds for dealing with data semantics, consistency and redundancy problems
  • To enable the growth of set-oriented data manipulation languages.

Go to the link to view the Codd paper.      

  • E. F. Codd, A Relational Model of Data for Large Shared Data Banks. (June, 1970). IBM Research Laboratory.

Common Terminology

The relational data model is based on the mathematical concept of a relation; this is physically represented as a table.

As Codd was a trained mathematician, the terminology was taken from mathematics, notably in the areas of set theory and predicate logic.

You will now be presented with the common terminology along with the structural concepts of the relational data model. Please be sure to click on all of the tabs:

Relation

  • A table with columns and rows.
  • This will be applicable to the logical structure of the database in terms of the external and conceptual levels of the ANSI-SPARC architecture.
  • This will not be applicable to the physical structure (internal level) of the database.
    • An additional alternative name is “file”.
    • In the figure below, Branch and Staff are relations (i.e. tables)

Attribute

  • This is a named column within a relation, i.e. inside a table.
    • If a table possesses 8 attributes, it simply means there are 8 columns.
  • An alternative name is “field”.
  • In the figure below, "branchNo", “street” and “postcode” are attributes in the Branch relation (table) whilst “fName” and “position” are attributes (columns) in the Staff relation.

Domain

  • This is the set of permissible values for one or more attributes.
  • It is a very important concept because it permits the user to define in a central location the meaning and source of values that attributes can hold. Examples in reference to the below figure:
    • The branchNo attribute can be a character-type with a size of 4 characters and a fixed range from B000 to B999.
    • The city attribute can be a character with a set maximum length (i.e. number of characters) of 20.
    • The postcode attribute can be a character with a set maximum length of 9.
    • The salary attribute can be a numeric value with a currency format (e.g. £xx,xxx.xx) and a defined range (e.g. £40,000.00 to £70,000.00).
    • The DOB attribute can be a date type with a range from 01-Jan-1920. The format can also be defined (e.g. dd-mm-yyyy).
  • It is important that when data is inserted into the database, it conforms to the specified domains of the respective attributes. This an important "integrity constraint".
    • For example, you cannot insert a person’s first name into the salary attribute because the attribute is expecting numbers and not letters.
  • There may be a request to multiply two attributes together:
    • This will only be successful if the two attributes are of the same domain, in this case a numeric type (integers or double).
    • If one attribute contains numbers in numeric type and another contains numbers in character type, the latter will need to be converted to numeric type before the multiplication takes place. This is permissible during a query.
  • An alternative name is "data type" (or "datatype").

Tuple

  • This is a row of a relation (i.e. a row in a table).
  • In reference to the figure below:
    • Each tuple (row) in the Branch relation contains four values, one for each attribute.
    • Each tuple in the Staff relation contains eight values.
  • The rows can appear in any order in the relation (table), and does not affect the meaning or representation of the data in that relation.
  • An alternative name is “Record”.
  • In the below figure, there are five tuples (rows) in the Branch relation (table) and six tuples in the Staff relation.

Degree

  • This is simply the number of attributes (columns or fields) in a relation.
    • If a relation contains six attributes (columns) then it is known as a six-tuple, and be classed as a “degree six”.
    • If a relation contains only one attribute it would be classed as a “degree one” and be called a unary relation (or one-tuple).
    • If a relation contains only two attributes   a binary relation.
    • If a relation contains only three attributes   a ternary relation.
    • If a relation contains more than three attributes   n-ary relation.
  • A degree of a relation is also a property of the intension of the relation.
    • The intension is derived from the structure of the relation along with the specified domains. This remains constant as long as the relation does not change.

Cardinality

  • This is the number of tuples (rows) in a relation (table).
  • This tends to change as data values are inserted or removed from a relation.
  • This is the property of the extension of the relation, and can be ascertained from the instance of the relation at a given moment.
    • In the below figure,  the Branch relation has a cardinality of five.

 

 

Keys and Schemas

The relational database is therefore a collection of relations (tables) with distinctive names (e.g. Branch as in the figure below). Be sure to click on all of the tabs below to learn about three very important terms when relating the tables within a database:

Primary Key (PK)

  • This is a unique identifier. It consists of one or more columns that uniquely identify each tuple in a relation.
    • An example of a unique identifier is the National Insurance (NI) number of a person.
  • PKs play an important role in the design and use of a database.
    • They can be seen as “look up values” for the rows of a table.
  • When a new relation (table) is being created, it is mandatory to designate a Primary Key, and to ensure there is a value in each row (e.g. the NI number).
    • A typical constraint here is that NULL values are not allowed.
  • As PKs are unique identifiers for a given record (row), they cannot be changed.
  • PKs are central to managing the relationships between tables.
  • In the figure below, the branchNo attribute in the Branch relation is a Primary Key.

Foreign Key (FK)

  • This is a reference to a specific row containing the primary key (e.g. the branchNo attribute in the Staff relation in the figure below).
    • It can act as a cross-reference between two or more relations.
  • The FK is in essence a “recording” of a PK, and is employed as a pointer to a different (or “foreign”) row typically in another table.
    • It is an attribute in one relation and a Primary Key in another relation.
  • FKs are not required to be unique inside a table. This is due to the fact that there could be multiple rows in one table containing the same value but which refers (or points) to the same unique value in another table.
    • In the figure below, you can see that the Staff relation contains three rows with the same branchNo attribute: B003.
    • This is known as a “one-to-many” relationship.

Candidate Key

  • This is actually a Super Key that contains no repeated attributes.
  • The Primary Key is usually selected from this.
  • It is possible for a relation to possess more than one candidate key, but only ever one primary key.
  • An example is when a table contains three of the following attributes: ‘NI Number’, ‘Employee ID Number’, and ‘Employee Surname’:
    • The first two attributes are candidate keys because one person can only have one NI number, and the “Employee ID Number’ is unique to one person.
    • There can however be more than one “Smith” in ‘Employee Surname’

Composite Key

This assists in uniquely identifying the record (row) in a relation by combining two or more attributes (columns) so as to ‘hone in’ on the unique record. neither attribute is required to be a foreign key.

  • If one of the attributes was a foreign key, we would attain a 'compound key'. 

Surrogate Key

  • This can be generated in the absence of any Primary Keys.
  • It does not alter the meaning of the relation.
  • It usually takes the form of an integer.
  • It can be generated just before a record is inserted into a relation.
  • Example:
    • If a relation consists of employee names, and dates of vacation for the financial year, but no ‘Employee ID Number’ or ‘NI Number’, then a surrogate key can be employed to uniquely identify the employee (e.g. V001, V002, etc.).

Relation schema

  • This is where the named relation is defined by a set of attribute and domain name pairs.
  • This can be considered to be a set of mappings from the attribute names to their corresponding domains.
    • This means that each element in the tuple (row) – where an element is analogous to a Microsoft Excel worksheet cell (e.g. C5, F32 or AB27) will consist of an attribute and contain a value for that attribute.
  • One of the tuples in the Branch relation can be expressed as follows: {(branchNo: B005, street: 22 Deer Rd, city: London, postCode: SW1 4EH)}.
    • This is further referenced as a relation instance because this is the situation at this moment in time. In a months’ time, the branch office could have relocated.
  • When we have a set of relational schemas that each contain a distinctive name, e.g. Branch, then we have a relational database schema.

So you may ask what is the actual relationship here shown in in the figure above?

Well we can say that there is a relationship between the Branch and Staff tables, notably that the branch office must have staff working in it from the manager down to the assistant.

Although there would appear not to be a physical link between these two tables, we can see that the two tables have the same column (“branchNo”) and can deduce that a relationship exists.

You may next be inclined to ask: Why not just have one big table?

This would be impractical because there would be duplication of information (introducing redundancy in to the database) which would lead to a waste of storage space.

In addition to this, it would make the database difficult to manage because if an item of information needed to be updated, then all the corresponding duplicate information would also need to be updated, for example the mobile phone number of a customer.

It is therefore more conducive to use separate tables which hold information based around a particular theme, as shown in the figure above for Staff and Branch.

What observations in relation to the relational data model can we make?

  • The name of one relation (table) is distinctive from all the other names in the relation schema.
  • Each element (cell) of a relation contains exactly one atomic (i.e. single) value.
    • There should not be two entries in one cell.
      • One branch address cannot have two postcodes.
  • Each attribute inside a relation has a distinct name.
  • The values of an attribute are all from the same domain.
  • Each tuple is distinct, i.e. there are no duplicate tuples.
  • There is no significance in how the attributes are ordered.
  • There is no significance in how the tuples are ordered.
  • It must be noted that the order could affect how efficiently the tuples are accessed, especially if there are thousands of them.

 

And finally, in relation to the figure above, an alternative way to express the relational (database) schema is to list the name of the relation followed by the attributes, for example:

  • Branch (branchNo, street, city, postCode)
  • Staff (staffNo, fName, lName, position, sex, DOB, salary, branchNo)

Note that the Primary Keys (branchNo in Branch and staffNo in Staff) have been underlined.

The Primary key must not contain any null values.
