Recovery Services

 

A DBMS must be able to restore a database to a former consistent state following a failure of some kind, whether that be software, hardware or both. There must be in place some form of “recovery service” that can enable the restoration.

Recall that non-database processing usually involves operations taking place on variables that are stored in memory. This is considered to be “volatile storage”. If there is a loss of power, then that storage is lost, as are any operations and updates to variables that were stored in memory.  

In terms of “non-volatile storage”, data contained in tables are stored on physical hard drives, but also likely replicated on other physical hard drives. In the event of a hardware failure, such as hard drive failure, the table(s) can be retrieved from other working hard drives, and the database restored.

Examples of failures include:

  • System crashes (which can involve loss of memory, i.e. the “volatile storage”).
  • Media failures such as hard drives.
  • Application (software) errors.
  • Natural physical disasters.
  • Carelessness by users leading to (unintentional) destruction of the data.
  • Sabotage!

The DBMS should be able to provide the following facilities in order to assist with the recovery of the database:

  • A backup mechanism that can make periodic backup copies of the database.
  • Logging facilities in order to keep track of the current state of the transactions, and the changes being made to the database.
  • A checkpoint facility in which any updates currently being made to the database are made permanent.
  • A recovery manager that will permit the system to restore the database back to a consistent state should a failure occur.

Recovery services are therefore very important in database management.
