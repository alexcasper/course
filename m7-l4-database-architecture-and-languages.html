 Database Architecture and Languages

 

A database system (a DBMS along with physical files and related application software) should be able to provide registered users an abstract view of the data in which certain details on how the data is stored and manipulated are hidden from them.

For example, users do not need to know the physical database storage details. 

 

Database Architecture

When designing a database, the general description and information requirements of the business organization must be considered and be correctly represented within the database.

Given that the database will be a shared resource, each user may require a different view of the data held in the database.

In order to satisfy these requirements, the architecture of the database management system (DBMS) should conform to the “ANSI-SPARC” architecture, which contains three key levels (click on the tabs to learn more):

External

  • The registered users’ views of the database.
  • A description of that section of the database that is most relevant to a particular user or group of users.

Conceptual

  • Can be considered to be a ‘community view’ of the database by describing the type of data stored in the database and the corresponding relationships among the data.
  • It can provide a mapping and independence between the external and internal levels.

Internal

  • This is the physical representation of the database on the PC server or mainframe.
  • This is where and how the data is stored using the data structures and file organisations.

 

ANSI stands for "American National Standards Institute".

SPARC stands for "Standards Planning and Requirements Committee".

Database Languages

Data access language has two parts. 

Data Definition Language (DDL) Data Manipulation Language (DML)

The DDL is used to specify/define the database schema including: 

  • Data types,
  • Structures and
  • Any Data constraints.

All appropriate specifications are thus stored in the database. 

 

 

The DML is used to both read and update the database including: 

  • ‘Querying’ and thus retrieving of the data 
  • Updating of the information within the database including 
    • Deletion of items,
    • Addition of new items, or 
    • Updating of current items

The DDL and DML are typically known as sub-languages because they can be called by a ‘host language’ in order to retrieve and process data from a database.

For example, a ‘host language’ program written in C++, Java, Python or R, may contain embedded SQL code, or a function/module to permit interaction with a database server.

Any conditions or iterations are carried out in the ‘host language’ program leaving the SQL code to focus on solely interacting with the database, and return the output (results) to the calling program.

We can extend the DML to two sub-topics: procedural and non-procedural. Please click on the tabs below to learn more.

Procedural DML

This permits the user to instruct the system what data is needed, and exactly how to retrieve that data.

The instructions must be clearly ‘understood’ by the system in order to return to the user exactly what data/information was requested.

Non-Procedural DML

This permits the user to instruct the system on what data is needed.

The user does not need to specify on how to retrieve it.

The DBMS translates the DML statement into a series of procedures that will internally retrieve the requested information from the database.

The user is not concerned with any data structures since the DBMS will contend with them in the background, and therefore the user attains a considerable degree of data independence.
