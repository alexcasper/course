 Types of Databases

 

There are different types of databases, with the most popular being the relational database.

Why? Because relational databases:

  • Provide us the ability to understand the way the data is related
  • Have a dedicated language that can efficiently query and therefore extract the required information, no matter how complex that query is

Therefore this course will primarily focus on relational databases.

The different types of databases, can be categorized into the following “generations”. Please be sure to click on all of the listed generations below to learn more.

 

First Generation (1960s)

It has been proposed that the use of a database management system (DBMS) came about during the NASA Apollo space missions of the 1960s because there was nothing available to handle the vast amounts of expected information the missions were going to generate.

We therefore saw the emergence of two approaches:

  • Hierarchical: This involves combining a multitude of small components into many larger components, which themselves are combined into a few larger components, and so on, until one final huge component remains.
  • Network: This enables the representation of more complex relationships than a hierarchical DBMS could model.

However first generation databases had a number of disadvantages including the writing of complex programs in order to answer what could be the simplest of queries, and minimal data independence.

Second Generation (1970s)

Relational database (RDB): The relational model provided a practical implementation of data structures and operations resulting in: 

  • The conception of structured query language (SQL) which is now the standard language for querying RDBs.
  • The first commercial products appeared in the late 1970s and early 1980s some of which were developed by Oracle and IBM.

The relational model is however limited in its modelling capabilities. This led to the development of the entity-relationship model (E-R) and other more complex database applications.

Third Generation (1990s)

We saw the development of: 

  • Object-Relational database (ORDB)
  • Object-Oriented database (OODB)

Third generation databases make use of dimensional databases which store data from several data sources, which are maintained from different business units of an organisation.

Cloud Technologies (2010s)

Current cloud technologies take full advantage of the web-database integration with the likes of Amazon Web Services (e.g. RDS or EC2) and Microsoft Azure (e.g. DataBricks).

Advantages and Disadvantages of the DBMS

Here are the key advantages and disadvantages of the DBMS. Be sure to click on all of the tabs below to learn more.

Advantages

A1. Control of data redundancy

  • Avoids multiple copies of the same information being stored.
  • May duplicate some information when appropriate in order to model more efficiently and thus increase performance.

A2. Data consistency

As multiple copies are minimized, it is easier and more efficient for the system to be consistent when data items are modified.

A3. Increased information from the same amount of data

In comparison to a file-based system in which key information can be easily missed.

A4. Sharing of data

The data in a database is available to all registered users within the business, whereas the data in a file-based system will be owned by the end-user or department (e.g. Contracts) and unavailable to other departments within the business.

A5. Improved data integrity

Ensures the validity and consistency of the data stored in the database by use of constraints which ensure that no violations take place by adhering to the ‘consistency rules’.

A6. Improved security

Prevents unauthorized access to the data. The DBMS enforces any security measures. Registered users will have a username and password in order to gain access to the database.

The users can also be restricted in what information can be accessed, meaning that some registered users who have seniority may have more access to the data than a registered user in a junior position.

A7. Enforcement of standards

This assists in the seamless exchange of data between systems following the necessary standards whether they be departmental, organizational, national or international.

These standards ensure the data are in the agreed formats following naming conventions, documentation standards, update procedures and access rules.

A8. Economy of scale

Cost savings can very likely be substantial if there is only one database set up for the entire business compared to the costs required to fund every department within the same business to maintain their respective file-based systems.

A9. Balance of conflicting requirements

The needs of one user or department may conflict with the needs of another user.

The database administrator (DBA) can ensure that the database provides the best use of the resources for the business as a whole ensuring optimal performance for important applications.

A10. Improved data accessibility and responsiveness

The data stored on the database will cross the departmental boundaries, and can thus be accessible to all registered users.

Dedicated languages like SQL can be used to access information from the database.

A11. Increased productivity

The user or programmer is more focused on the task at hand and does not need to be concerned with where certain bits of information reside, or have to construct a complicated application to extract the information therefore saving time, cost, and leading to increased productivity.

A12. Improved maintenance through data independence

A DBMS separates the data description from the application which results in the application being immune to changes made to the data description.

This is superior to the file-based system in which the description of the data and the logic for accessing the data are ‘hard-coded’ into each application program, and thus making the program dependent on the data. 

A13. Increased concurrency

If two registered users are updating an item of information with different values, a DBMS will ensure such a conflict does not occur and so data integrity is not lost, as would be the case in a file-based system.

A14. Improved backup and recovery services

A modern and efficient DBMS will provide the capability to minimize the amount of processing that is lost should a failure occur by regular backups of the database, along with logs to track the transactions taking place.

 

Disadvantages

D1. Complexity

A very good DBMS will likely be an extremely complex piece of software. All users (DA, DBA, etc.) will need to be aware of this, as in being unaware will likely lead to bad design decisions which could prove costly for the business.

D2. Size

Given the complexity and wide breadth of functionality of the database will make it a very large piece of software occupying hundreds of megabytes of disk storage and memory.

D3. Cost of the DBMS

This varies significantly depending on the environment and functionality provided.

  • A single-user DBMS for a personal computer may only cost £100.
  • A large frame multi-user DBMS servicing hundreds of registered users may cost from £100,000 to £1m. There is also the maintenance cost to add on top!

D4. Additional hardware costs

On large systems that occupy multiple hard drives (that have been RAID-ed) additional costs will be incurred in replacing hard drives that fail, or to expand the size of the DBMS.

Consequently, it may be necessary to purchase a dedicated high-spec machine to host the DBMS.

D5. Cost of conversion

It could be costly to convert the current systems in place within an organization in favour of a DBMS because staff will also need to be trained to efficiently use the DBMS. 

D6. Performance

An application accessing a DBMS may not be as efficient compared to an application accessing a file-based system to carry out a specific task because the DBMS caters for many applications.

D7. High impact of a failure

As all the resources are centralized, this can increase the vulnerability of the DBMS.

If some components of the database fail, this could affect a multitude of registered users, and therefore lead to a halt in operations.
