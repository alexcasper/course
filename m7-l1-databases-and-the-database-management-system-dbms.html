 File-Based Systems


Prior to the advent of databases, there were paper-based and then computer-based filing systems.

In terms of the latter, these were seen as a collection of application programs which perform services for the end users, such as the generation of subject-specific reports.

Each program would be seen to define and manage its own data. However, they were unable to interact with each other.

The figure below illustrates the concept of a file-based processing system.

 

The “Sales files” and “Contracts files” are physically stored in distinctive locations, and are unable to interact with each other. This means that if information in the “Sales files” such as ‘client address’ is updated, the same information contained in the “Contracts files” will not be automatically updated.

 Let's take a look at the contents of the Sales and Contracts files:

Sales Files

     PropertyForRent (propertyNo, street, city, postcode, type, rooms, rent, ownerNo)

     PrivateOwner (ownerNo, fName, lName, address, telNo)

     Client (clentNo, fName, lName, address, telNo, prefType, maxrent)

Contracts Files

     Lease (leaseNo, propertyNo, clientNo, rent, payMthd, deposit, paid, rentStart, rentEnd, length)

     PropertyForRent (propertyNo, street, city, postcode, rent)

     Client (clientNo, fName, lName, address, telNo)

If we compare the Client files in both systems, we see that the Client file in Contracts does not contain the prefType and maxrent attributes.

This implies that a program that was designed to extract information from the Client file in Sales will need to be adapted (or “made aware”) that the prefType and maxRent attributes are missing in Contracts. This could cause the program to ‘crash’, and therefore lead to costly delays.

Conversely, a program accessing the attributes from the Client file in Contracts will be unaware of the prefType and maxRent attributes in the Client file of Sales, and so the user could be missing out on important and vital information, which again could be costly to the business.

There are obvious limitations to employing a file-based approach. Let's look at some limitations (click on the tabs):

Separation and Isolation of Data

Each program is maintaining its own set of data, and users of one program may be unaware of potentially useful data being held by other programs.

Duplication of Data

The same data (i.e. information) is held by different programs, which results therefore in wasted storage space in conjunction with the likelihood of different values and/or different formats for the same item/attribute.

Data is Dependent

Dependent on how the file structure is defined in the coding of the program.

Incompatible File Formats

Programs are written in a variety of different languages making it difficult to access the files pertaining to the other programs in use.

Task Oriented

Programs tend to be designed to perform a particular task/action (e.g. queries are always the same) and so any new requirement will involve the necessity to adapt the current program, or to construct a new program entirely.

 

Therefore in file-based systems the data definition was dependent upon the instruction given by the application programs and there was no control over the access and manipulation of the data due to the enforced restrictions by the application programs.  

 

Database Management System (DBMS)

A much improved approach was required which resulted in the conception of the database, and thus a database management system (DBMS).

A database can be described as:

  • Shared collection of logically related data, which includes a description of this data, designed to especially meet the information requirements of a business.
  • It contains a system catalogue (i.e. “metadata”) which provides descriptions of the data in order to enable program-data independence.
  • It contains logically related data made up of entities, attributes and relationships pertaining to the business organisation’s information.

The database management system (DBMS) is the software that is employed to define, create, maintain, control user access and interaction with the database. A schematic of a DBMS is shown below.

Functions of a Database

An efficient DBMS must be able to handle the following circumstances:

  • Manage the physical structure of the database in terms of physical storage.
    • The database could be stored on one large hard drive in a dedicated PC server, or spread across multiple hard drives (e.g. via RAID5) in a dedicated PC server, or spread over numerous dedicated PC servers.
  • Control user access to the database by adhering to the privileges assigned to the user, therefore ensuring the security of the database and the contents held within.
  • Handle multiple users at the same time, and ensure no conflicts ensue.
  • Periodic backups of the data within the database, and to permit recovery options should the database fail or “suffer damage”.
  • Make use of a dedicated language to allow access to the database – a ‘data access language’.
  • To ensure the integrity of the data by enabling constraints and periodic checks on the data.
  • To make use of a ‘data dictionary’ which will maintain information about the structure of the database.
