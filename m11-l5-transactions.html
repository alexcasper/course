 Introducing Transactions

 

So far in this course you have been looking at examples of SQL commands that are entered into a database management system (DBMS) application mostly in SQLite3 or Oracle Live SQL.

You, the single user, have also entered commands into these systems which have resulted in retrieving, updating or deleting information from a table in a database.

This can therefore be be seen as a transaction. 

A transaction, however, is not limited to ‘one action’, for example, to “update the bonus to be paid to employee number 2442”.

A transaction can actually involve ‘numerous actions’. This means that a specific request can generate ‘many actions’ leading to the updating of multiple tables within a database, or numerous tables across many databases!

This could be initiated by a database administrator, or by an application where one user ‘points-and-clicks’ the options from a dropdown menu, enters some values where appropriate, and then selects ‘Go’. 

At the same time, there could be many users accessing the same database(s), and retrieving information or making changes to information within the same table(s).

This therefore poses a risk in that one user could extract the wrong/incorrect value for a client because another user has just changed this value (possibly incorrectly!).

Therefore the DBMS must ensure that the database remains consistent (i.e. concurrent) and reliable for all users. 

In this regard, the DBMS must be able to provide the following functionalities:

  • Transaction support
  • Concurrency control services
  • Recovery services

 

Concurrency Control and Recovery Services

It is important to stress that the three functionalities listed above are actually mutually dependent.

For example, concurrency control services and recovery services are vital for protecting the database against inconsistencies and loss of data.

Since DBMSs permit multiple users to carry out simultaneous operations on the database, if there are no control mechanisms in place, then the access by the users will very likely interfere with one another. This is where the concurrency control services ‘step in’. 

Concurrently, the recovery services involves restoring a database to a correct stable state following a failure.

This failure could be a result of hardware, such as PC server(s), network(s) and/or hard drive(s) failure, or any software error such as an application crash, logical flow error in a program/application, or even erroneous SQL coding that leads to corruption of the data within the database(s).

Either way, the DBMS must be able to restore the database(s) to a consistent state.

'Recovery services' and 'Concurrency control services' will be discussed in greater detail in Part 2 of the course.
