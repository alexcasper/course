 Database Views and Roles

 

Database Views

You now have a basic understanding of databases and database management systems. But what about the users of a database and/or DBMS?

Depending on the size of the database, different users will have different “views” of the database. We will talk more about “views” later in the course but for now simply think of the term as a subset of the database. “Views” assist with the following:

  • Reducing the complexity in understanding the ‘schema’ or layout of the database.
  • Provide a level of security in that the original items within the database are protected.
  • Customization of the appearance of the database.
  • Maintain a consistent view of the database for the end user, even if there are underlying changes made to the database itself.

 

Specific User Roles for Employing a DBMS

So who are the database end users? You, for one!

Let’s look now at some specific roles in DBMS. Please be sure to click on all of the roles shown below (or just 'Expand All'):

Data Administrator (DA)

  • Assists with the conceptual and logical design of the database.
  • Manages data resources including the planning of the database.
  • Develops and maintains the standards, policies and procedures.
  • Ensures database development supports corporate objectives.

Database Administrator (DBA)

  • Assists with developing the physical design of the database involving its implementation, security and integrity control.
  • Maintains the operating system that the database sits in.
  • Ensures users experience satisfactory performance.
  • Has detailed knowledge of the system environment and the DBMS.
    • The DBA role is seen as more technically demanding compared to a DA role.

Database Designers

  • These tend to be split between the “logical” and “physical” database designers.
  • The logical database designer can be tasked with identifying:
    • Data attributes and entities
    • Data relationships and any constraints or “business rules” to be included 
    • Suitable database models
  • The physical database designer can be tasked with deciding how: 
    • The logical database design is to be physically realized, for example, whether the database will sit on one server PC or distributed across many
    • To map the logical database design into a set of tables and integrity constraints.
    • To select specific storage structures and access methods to ensure an efficient performance

Application Programmers

  • Application programmers develop the user interface with the required functionality for the end user.
    • The programmers may employ third generation languages (3GL) such as C++, Java or Python.
  • These interfaces contain fixed query statements that ask the DBMS to perform a specific task/operation on the items of data within a database such as to retrieve, insert or update content.
    • Consider the self-service cashier at a supermarket which has just scanned a bottle of wine that a customer intends to purchase.
    • The interface (in this case the program that is in use) asks the DBMS to identify from the database the item from the barcode, and then to retrieve the current price to display to the customer, and to also log the intent to purchase.
    • At the same time, because alcohol has been identified as a product being purchased, an event is triggered due to an established constraint within the DBMS advising the customer to seek approval from the attendant in order to proceed with the purchase of the bottle of wine.

Other End Users

  • These are the ‘clients’ of the database. Typically the database will have been designed and implemented, and then maintained to meet the (business) needs of the users.
  • Such end-users can be “Naïve” or “Sophisticated”.
    • Naïve end-users will be oblivious to the presence of the DBMS and access the database via specialized user interfaces (e.g. the supermarket checkout).
    • Sophisticated end-users tend to possess a very good understanding of the structure of the database along with the facilities available from the DBMS. These users may also be Application Programmers.
