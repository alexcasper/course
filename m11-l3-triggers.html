 Establishing a Trigger

 

A trigger can be set up to define an action that the database must undertake should an event occur which can compromise data within a table, some tables or the entire database.

A trigger can also be used to enforce some referential integrity constraints, to enforce complex constraints, or to audit changes to the data.

Triggers are said to be based on the Event-Condition-Action (ECA) model (please be sure to click on the tabs below):

Event

  • The event(s) can involve the following SQL commands applied to records within a table or tables or views: INSERT, UPDATE or DELETE.
  • The event(s) can involve the following SQL commands applied to any object within a schema such as tables, sequences and indexes: CREATE, ALTER or DROP.
  • The event(s) can also include the startup and shutdown of a database, or when specific users log on and log off the database.
  • The event(s) can also include any error messages that are generated.

Condition

  • This determines whether the action should be executed or ignored.
  • The condition is an optional parameter, and so does not need to be specified.
    • If the condition is specified, the action will only be executed if the condition happens to be TRUE.

Action

This contains the SQL statements and code to be executed when a triggering statement is issued and the trigger condition (if used) evaluates to TRUE. 

The trigger can be directed when to take effect in relation to the event: BEFORE, AFTER or INSTEAD OF.

The latter can be useful for modifying views that cannot be altered using the normal SQL data manipulation language (DML) statements. In this instance, the trigger gets fired in place of executing the original SQL statement. 

Two types of triggers that take place are (please be sure to click on the tabs below):

Row-Level Triggers

  • These occur for each row of the table.
    • The SQL keywords used are: FOR EACH ROW.

Statement-Level Triggers

  • These occur only once for each statement and can affect multiple rows in a table.
  • The SQL keywords used are: FOR EACH STATEMENT

Triggers can also activate themselves sequentially. This can occur should there be a situation where one trigger executes, and on completion of altering the database, a second and other triggers activate, which themselves cause more changes to the database, and possibly activate even more triggers, therefore leading to a cascade of triggers!

Advantages and Disadvantages of Triggers

 

Advantages of Triggers

The advantages of establishing triggers are (please be sure to click on all of the tabs below):

Elimination of Redundant Code

The trigger only needs to be stored once in the database and not on every user’s profile.

Simplifying Modifications

  • If a trigger is changed, it only needs to be done once as all the applications automatically employ the updated trigger.
  • The triggers are therefore only coded once, tested once, and finally enforced centrally in order for all of the applications to have access.
  • The database administrator (DBA) with advanced knowledge can control efficiently, and if necessary, audit the trigger.

Increased Security

As the triggers are stored in the secured database, they are therefore protected.

Improved Integrity

Triggers that are stored in the database can consistently enforce integrity constraints across all of the platforms by the database management system (DBMS).

Improved Processing Power

The triggers actually add processing power to the DBMS and to the database.

Good Fit With the Client-Server Architecture

As the trigger is stored “centrally” within the database, it can be executed by a user leading to automatic procedures of sequences of checks and operations by the database server itself.

  • This increases performance because there is no need to transfer operations across the network since the triggers are already on the database (server).

Disadvantages of Triggers

The disadvantages of establishing triggers are (please be sure to click on all of the tabs below):

Performance Overheads

Managing and executing the triggers results in a performance overhead which needs to be balanced against the “improved processing power” mentioned previously.

Cascading Effects

There is a risk of one trigger firing off a number of subsequent triggers in a “cascading manner”.

  • This can cause a significant change to the tables in the database.
  • It can be hard to foresee the overall effects when the trigger is being designed.

Cannot be Scheduled

The trigger occurs when the event it is based on actually happens.

Less Portable

Because most of the DBMSs have their own specific way of dealing with triggers, this affects portability of the database into other systems.
