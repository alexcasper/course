 Tables

Tables are incredibly important to understanding how many information systems function. Tables underly both relational databases and spreadsheets. While it is possible to have a non-relational database, or to use a spreadsheet in a way other than intended, understanding how to use tables correctly will be incredibly valuable for your work with data management systems. ServiceNow, despite its additional features and functionalities, has at its heart a collection of preconfigured tables. These tables each store information about an individual entity - like a record of a particular server's configuration - or an event - such as an incident that needs to be dealt with. To understand what is in these tables, lets look at an example below.

Rows and Columns

Here we have an example of a row. This dataset appears similar to something we might find in ServiceNow or a relational database, but it's actually just a a spreadsheet. Here, a row represents an individual - in this case, an employee of an IT company. By following the row horizontally, we can find the position, hire date, team assignment and manager. We call this row a record. The row has a unique identifier, or key, on the left. 

 Below is the same information, but instead of highlighting a row, we are looking at a column. We can see that if we pick a particular column or field, we can get an amount of comparable information. From this field we could, for example, find out how many sales executives there are employed by the company. Depending on the type of information system, what you enter into a column will be more or less restricted. An advantage of more advanced platforms is that they will force users to put a specific data type in a field, preventing us from entering data in incompatible formats.

 

Lookups

We mentioned the importance of keys in the previous section. A key can be used to relate one piece of information with another. For example, the last column, reports_to, is a whole number (integer). Without any more information, this is pretty useless. Yet if we are told that the reports_to column is a reference to another record, we can now find information about that individual's line manager. We can lookup that other record and retrieve the other information, either using SQL in the case of databases, or automatically in the case of ServiceNow. 

Lookups can and often do occur between tables. In fact, columns H and I in the above diagram are actually being pulled elsewhere. We can be told to look up column G in an entirely different table, and populate columns with the information we retrieve. While we are not going to explore this dataset further here, you can see that each team is a part of another team (a reference to the same table) and has a leader (a reference back to the employee table).
