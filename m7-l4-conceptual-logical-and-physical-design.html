Data Models Reading

The storing, accessing and updating of data in a relational database is defined by both the data model and the structure of the database. The three stages to account for when designing the model are shown in the tabs below. 

It is essential to click on all of the tabs to learn more.

Conceptual Design 

This should be independent of all physical considerations.

  • This tends to be non-system specific presented at a higher level to business stake holders.
    • For example, this will not mention primary keys and foreign keys.

The recommended steps for building the conceptual data model are:

  1. Identify the required entity types.
  2. Identify the important relationship types.
  3. Identify and associate the attributes with either the entity or relationship types.
    • This involves associating the attributes with the appropriate entity or relationship types and to document the details of each attribute.
  4. Determine the attribute domains in the data model, and to document the details of each domain.
  5. Determine the candidate, primary and alternate key attributes.
    • This involves identifying the candidate keys for each entity, and if there are more than one candidate key then to assign one to be the primary key, and the others to be the alternate keys.
  6. Depending on the task, consider whether the use of enhanced modelling concepts will be applicable.
    • This would involve considering the concepts of specialization, generalization, aggregation or composition.
  7. Check model for redundancy.
    • If found, omit it.
  8. Validate the conceptual model against user transactions.
    • This is important to ensure that the conceptual model does support the required transactions.
  9. Review the conceptual data model with the user.
    • This is to ensure that the model is indeed a “true” representation of the data requirements and needs of the business organization.

Logical Design 

This will be based on a specific model, for example relational or object-oriented, but independent of a particular DBMS and other physical considerations (e.g. storage):

  • This will translate the conceptual model into the relevant structures that will permit implementation into a DBMS.
    • For example, ERDs showing relationship types, primary and foreign keys will be produced. 

In terms of the relational model, the recommended steps for building the logical data model are:

  1. Derive relations for the logical data model.
    • This includes creating relations in order to represent the entities, relationships, and attributes that have been identified.
    • This can involve identifying strong and weak entity types, and whether relationships are one-to-one, one-to-many or many-to-many.
      • A strong entity always has a primary key and is not dependent on the existence of another entity.
      • A weak entity does not possess a primary key and is usually dependent upon a strong entity for it to exist.
  1. Validate the relations using normalization.
    • The objective of normalization is to ensure that the given set of relations (tables) possess a minimal yet sufficient number of attributes that are necessary to support the data requirements of the business organization.
    • Normalization will be discussed in more detail in Part 2 of the course.
  2. Validate the relations against user transactions.
    • This is to ensure that the transactions can be supported by the relations within the logical data model.
  3. Define the integrity constraints.
    • This is to ensure that the database does not become incomplete, inaccurate and inconsistent.
  4. Review the logical data model with the user.
    • This is to confirm whether the user(s) believe that the logical data model is a true representation of the data requirements of the business.
  5. Depending on the task, merge the logical data models into a global model.
    • The global model needs to reflect all of the user views of a database.
  6. Check for the potential for future growth.
    • The objective here is to determine whether there are likely to be any significant changes in the future, and whether the logical data model will be able to accommodate those changes.

Also see the following reading:

GIS&T Body of Knowledge, 2022. DM 35 Logical Data Models.

 Physical Design

This results in the production of a description on how implementation of the database is to take place on the “secondary storage”:

  • It will describe the base relations, the arrangement of the files, any indexing designs for efficient accessing of the data, along with any associated integrity constraints and security measures.
  • This incorporates the changes needed in order to attain the required performance.
    • For example, the ability to be able to insert, delete or update records via the application of functions.

A reminder that the logical database design is concerned with the “what” whilst the physical database design is concerned with the “how”. The physical database design incorporates the logical data model along with the documentation that describes that model.

In terms of the relational model, the recommended steps for building the physical data model are:

  1. Design the base relations.
    • For each relation, need to define:
      • The name of the relation.
      • Provide a list of simple attributes.
      • The primary keys, along with the alternate and foreign keys.
      • The referential integrity constraints for any identified foreign keys.
    • Consideration must also be given to the data dictionary which will contain for each attribute:
      • The domain that defines the data type (e.g. integer), length, and any constraints (e.g. cannot be negative).
      • An optional default value for the attribute.
      • Whether it can hold NULL values.
      • Whether it is derived, and if so, how it should be computed.
  1. Design representation of the derived data.
    • The logical data model and the data dictionary are examined in order to produce a list of derived attributes, which will either be stored in the database, or calculated when required.
      • The cheapest of the additional cost involved in either process is chosen subject to performance constraints.
  1. Design the general constraints.
    • Defining business constraints, for example restricting a staff member to a fixed number of simultaneous projects.

Factors in Designing a Database

In summary, to successfully design a database, the following key factors are very important:

  • The designer must work interactively with the users so that it is known exactly what is required. 
    • It might be necessary for the designer to prepare a questionnaire for the users to complete and then collate all the responses.
  • There must be a structured methodology throughout the data modelling process.
  • There must also be employment of a data-driven approach.
  • The incorporation of structural and integrity considerations into the data model is paramount.
  • The data modelling methodology must also combine the techniques of conceptualisation, normalisation and transactions.
  • Diagrams should be employed to represent as much of the data model as is possible.
  • There should also be the use of a database design language (DBDL) in order to represent additional data semantics.
  • A dictionary should also be constructed to accompany the data model diagrams.
  • The designer will need to be supportive for repeating any steps of the design process.
